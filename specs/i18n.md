# Internationalisation (i18n)

This is the master i18n reference for all SeeMaths games. Every game derived from this
template follows this architecture exactly. Game-specific specs (e.g.
`maths-distance-calculator/specs/i18n.md`) adapt this pattern for their own string
inventories but do not deviate from the structure described here.

**Files:**
- `src/i18n/types.ts` — `Translations` type + `TFunction` type
- `src/i18n/en.ts` — English strings (source of truth)
- `src/i18n/zh.ts` — Chinese (Simplified)
- `src/i18n/hi.ts` — Hindi
- `src/i18n/index.ts` — React context, hooks, helpers
- `src/components/LanguageSwitcher.tsx` — globe icon + dropdown
- `api/translate.ts` — Vercel serverless: on-demand OpenAI translation

---

## Design principles

### The golden rule

Every user-visible string — UI labels, buttons, game feedback, question prompts, PDF
content, email body — must go through `t()`. A hardcoded English string in any of
these locations is a bug.

This applies without exception to:
- All React component render output
- All question prompts (see pattern below)
- All PDF text (title, labels, objective, encouragement, footer)
- All email body strings
- All session report modal text
- All rotate prompt text
- All language switcher UI

### Question prompt translation pattern

Question prompts look like natural language sentences that include variable values
(names, numbers, place names). The challenge is that different languages have different
grammatical structures and word orders — you cannot just swap words into an English
sentence structure.

**The solution: template keys with `{placeholder}` substitution.**

Each distinct question sentence structure is a separate translation key. The template
itself is translated into each language (including word order), and the variable values
are substituted at render time using `{placeholder}` tokens.

Generic example:
- `"game.prompt.typeA"` — English: `"{subject} needs to go from {start} to {end}. What is the distance?"`
- Hindi: `"{subject} को {start} से {end} तक जाना है। दूरी क्या है?"`
- Chinese: `"{subject} 需要从 {start} 到 {end}，距离是多少？"`

The word order, verb placement, and sentence structure are entirely determined by the
translation string — the placeholder names just mark where the variable values go.

**How question generators must work:**

- All question generator functions accept `t: TFunction` as a parameter.
- They call `t("game.prompt.typeA", { subject, start, end })` to produce the rendered
  prompt string.
- The `prompt: string` field on the question object stores this pre-rendered,
  locale-specific string.
- This is "pre-baked" — the prompt is baked at generation time in the current locale,
  not re-translated on render.
- If the locale changes mid-session, new questions will be generated in the new locale;
  already-generated questions retain their baked locale.

**What must NOT be done:**
- Do not build prompt strings with string concatenation or template literals.
- Do not store the prompt template separately and render it elsewhere.
- Do not translate question prompts at render/display time (that would be real-time
  conversion, not pre-baked).

**Every new game must define translation keys for all its question prompt variants.**
There are no exceptions. If a question type exists, a template key must exist for it.

### PDF translation

`generateSessionPdf(summary, t)` accepts `t: TFunction` as a second parameter. Every
string drawn to the PDF — title, labels, objective text, encouragement messages,
footer — uses `t("pdf.someKey")`. No hardcoded English strings in the PDF generator.

### Email translation

`emailReport(summary, email, locale)` pre-translates all email strings using
`getT(locale)` before sending to the API. The `/api/send-report` serverless function
uses pre-translated strings when provided, falls back to English otherwise.

## Architecture

### Provider setup

Wrap the app root in `I18nProvider`:
```tsx
// App.tsx
import { I18nProvider } from './i18n'
export default function App() {
  return <I18nProvider><YourScreen /></I18nProvider>
}
```

### In React components

```ts
import { useT } from "../i18n";
const t = useT();
t("game.correct")                          // → "Correct! 🎉"
t("game.wrongAnswer", { answer: 5 })       // → "The answer was 5"
```

### Outside React (PDF generation, share report)

```ts
import { getT } from "../i18n";
const t = getT(locale);   // locale is a string like "en", "zh"
```

`getT` reads from `localStorage` for cached custom translations.

### Locale persistence

Active locale is stored as `localStorage["lang"]` (key `"lang"`).
Initialized from `localStorage` on mount, defaults to `"en"`.

---

## `TFunction` type

```ts
export type TFunction = (key: TranslationKey, vars?: Record<string, string | number>) => string;
```

Interpolates `{placeholder}` tokens. Unknown keys fall back to English (`en[key] ?? key`).

## `src/i18n/index.ts` — full API

### Context

```ts
interface I18nContextValue {
  locale: string;
  setLocale: (code: string) => void;
  t: TFunction;
}
```

### Exported hooks

```ts
export function useT(): TFunction
// Returns the translation function for the current locale.

export function useLocale(): { locale: string; setLocale: (code: string) => void }
// Returns the current locale code and setter.
```

### Non-hook helpers

```ts
export function getT(locale: string): TFunction
// Build a translation function for the given locale (for non-React code).
// Falls back to English for missing keys.

export function cacheTranslation(locale: string, translations: Translations): void
// Save a custom translation to localStorage (key: "i18n_cache_<locale>").

export function saveCustomLang(code: string, name: string): void
// Persist a custom language name mapping in localStorage (key: "i18n_custom_langs").

export function getCustomLangs(): Record<string, string>
// Load all previously translated custom language names from localStorage.

export function getIntlLocale(locale: string): string
// Returns the BCP-47 locale string for use with Intl APIs.
// e.g. "hi" → "hi-IN", "zh" → "zh-CN". Falls back to "en-AU".

export function getLocaleFormat(locale: string): LocaleFormatConfig
// Returns a full format config for the given locale, merging locale-specific
// overrides onto the default (en-AU) config. Use this when formatting dates,
// times, and durations in PDF and email output.
```

### `LocaleFormatConfig` interface

```ts
export interface LocaleFormatConfig {
  intlLocale: string;                          // BCP-47 tag for Intl APIs
  timeOptions: Intl.DateTimeFormatOptions;     // used in email body
  emailDateOptions: Intl.DateTimeFormatOptions;
  pdfDateOptions: Intl.DateTimeFormatOptions;
  compactDurationLabels?: {                    // overrides "m" / "s" labels
    minute: string;
    second: string;
  };
  useCompactDurationInEmail?: boolean;         // true → use compact labels in email
}
```

Default config (used for EN, ZH, ES, RU):
```ts
{
  intlLocale: "en-AU",
  timeOptions: { hour: "numeric", minute: "2-digit" },
  emailDateOptions: { year: "numeric", month: "long", day: "numeric" },
  pdfDateOptions: { weekday: "long", year: "numeric", month: "long", day: "numeric" },
}
```

Hindi overrides (`LOCALE_FORMAT_MAP["hi"]`):
```ts
{
  timeOptions: { hour: "2-digit", minute: "2-digit", hour12: false },
  compactDurationLabels: { minute: "m", second: "s" },
  useCompactDurationInEmail: true,
}
```

`getLocaleFormat` merges the locale's overrides onto the default, then sets
`intlLocale` from `INTL_LOCALE_MAP`. Both the short code (`"hi"`) and full
BCP-47 tag (`"hi-IN"`) resolve correctly via the base-code fallback.

### Built-in locales

```ts
export const BUILT_IN_LOCALES: Record<string, Translations> = { en, zh, hi };

export const LOCALE_NAMES: Record<string, string> = {
  en: "English", zh: "中文", hi: "हिन्दी",
};

// Maps locale code → BCP-47 tag used by getIntlLocale()
const INTL_LOCALE_MAP: Record<string, string> = {
  en: "en-AU", zh: "zh-CN", hi: "hi-IN",
};
```

**Built-in languages**

| Code | Language | Script | Intl locale |
|------|----------|--------|-------------|
| `en` | English | Latin | `en-AU` |
| `zh` | Chinese (Simplified) | Han | `zh-CN` |
| `hi` | Hindi | Devanagari | `hi-IN` |

### Interpolation

`{placeholder}` tokens in translation strings are replaced by values from `vars`:
```ts
t("game.tapScreen", { count: 2, total: 5 })  // → "Tap 2/5"
```
Unknown placeholders are left as-is: `"{unknown}"`.

## `LanguageSwitcher` component

Globe icon (🌐) button in the top-right toolbar cluster.

On click → opens a dropdown with:
- One button per built-in language (EN, ZH, ES, RU, HI)
- Any previously translated custom languages (from `getCustomLangs()`)
- "Other..." option to trigger on-demand translation

**On-demand translation flow:**
1. User clicks "Other..." → dialog asks for language name
2. `POST /api/translate` with `{ targetLang: "French", strings: en }` (all English strings)
3. Response: `{ translations: {...}, langCode: "fr" }`
4. `cacheTranslation("fr", translations)` + `saveCustomLang("fr", "Français")`
5. `setLocale("fr")` — dropdown now includes "Français"

## On-demand translation API (`api/translate.ts`)

**Endpoint:** `POST /api/translate`

**Required env var:** `OPENAI_API_KEY`

**Request body:**
```ts
{ targetLang: string, strings: Record<string, string> }
```

**Response:**
```ts
{ translations: Record<string, string>, langCode: string }
```

**Model:** `gpt-4o-mini`, temperature 0.3, `response_format: { type: "json_object" }`

**System prompt rules:**
1. Preserve all `{placeholder}` tokens exactly
2. Do not translate URLs
3. Do not translate brand names: SeeMaths, [Game Name], DiscussIt
4. Return `{ "translations": {...}, "langCode": "<ISO 639-1>" }`

## PDF and email language support

`generateSessionPdf(summary, t)` accepts `t: TFunction` as a second parameter. Every
string drawn to the PDF uses `t()` — no hardcoded English anywhere in the PDF
generator.

`emailReport(summary, email, locale)` pre-translates all email strings using
`getT(locale)` before sending to the API. The `/api/send-report` serverless function
uses pre-translated strings when provided, falls back to English otherwise.

jsPDF uses Helvetica (Latin) by default. For non-Latin scripts (Hindi, Chinese),
load and embed a Unicode font at PDF generation time using `loadFontBase64(locale)`,
then register via `doc.addFileToVFS()` + `doc.addFont()` + `doc.setFont()`.
Font files are served from `/public/fonts/` (NotoSansDevanagari for Hindi, ArialUnicode for Chinese).
`generateSessionPdf` accepts `locale` as a third parameter for this purpose.

## Adding new string keys (for new games)

1. Add the key to `src/i18n/types.ts` `Translations` type
2. Add the English value to `src/i18n/en.ts`
3. Add translations to `src/i18n/zh.ts` and `hi.ts`
4. TypeScript will error if any locale file is missing the key
